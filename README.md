# 🎯 AI Contact Hub

Backend-сервис для обработки обращений через форму обратной связи с интеграцией AI для анализа тональности сообщений.

Проект реализован на FastAPI с разделением на слои: domain, application, infrastructure, entrypoints.

Основной сценарий:

```
HTTP request
|
v
API endpoint
|
v
Validation (Pydantic)
|
v
Application use case
|
+--> Save contact (PostgreSQL)
|
+--> AI sentiment analysis
|
+--> Send email notifications
|
v
HTTP response
```
Проверка email адресов пользователей / регистрация пользователей может быть реализована при дальнейшем развитии проекта. В текущем бизнес-сценарии при ошибках, связанных с отправкой email пользователю, ошибки логируются, а в email-уведомление владельцу включается примечание, сгенерированное при перехвате низкоуровневого исключения SMTP-клиента.

Создать файл `.env`:

```env

PGADMIN_DEFAULT_PASSWORD=pgadmin_password
GROQ_API_KEY=you_groq_api_key
SMTP_PASSWORD=your_smtp_password
SMTP_HOST=smtp_host
SMTP_PORT=smtp_port
SMTP_USER=email_address_relating_to_smtp_settings
OWNER_EMAIL_ADDRESS=email_address_to_which_emails_for_owner_will_be_sent
```
## Запуск через Docker Compose

Сборка:

```bash
docker compose --build --no-cache
```

Запуск:
```bash
docker compose up -d
```
После запуска доступно:
- API  
http://localhost:8000/api/contact

- Swagger документация:  
http://localhost:8000/docs

## Паттерны проектирования
### Dependency Injection

Зависимости собираются через FastAPI Depends:
- database session
- Unit of Work
- repositories
- application services
### Repository Pattern

- работа с данными изолирована от бизнес-логики.
- application layer зависит только от интерфейсов

### Unit of Work

- управляет транзакциями БД
- позволяет контролировать commit / rollback


## Объяснение выбора технологий

### FastAPI

FastAPI выбран для реализации REST API благодаря:
- поддержке асинхронной разработки
- встроенной валидации данных через Pydantic
- автоматической генерации OpenAPI/Swagger документации
- удобному механизму Dependency Injection через Depends

### PostgreSQL

Используется для хранения обращений пользователей и результатов AI-анализа.

Выбор обусловлен:
- надежностью хранения данных
- поддержкой транзакций
- широким применением в backend-разработке
- хорошей интеграцией с SQLAlchemy

### SQLAlchemy + Repository Pattern + Unit of Work

- SQLAlchemy выбрана как ORM для работы с БД благодаря ее удобству, широким возможностям и полулярности в индустрии
- Repository Pattern позволяет отделить бизнес-логику от деталей хранения данных
- Unit of Work используется для управления транзакциями и контроля операций commit/rollback
- Такой подход упрощает тестирование и дальнейшее развитие приложения

### Alembic

Alembic используется для управления изменениями структуры базы данных. Выбран как наиболее простой и популярный инструмент для миграций.

Миграции позволяют:
- версионировать изменения схемы;
- воспроизводить состояние базы данных на разных окружениях.

### Docker / Docker Compose

Docker используется для создания воспроизводимого окружения разработки.

Docker Compose позволяет запускать приложение вместе с зависимостями (например, PostgreSQL) одной командой и уменьшает различия между локальной и серверной средой.

### AI API

AI-интеграция вынесена в отдельный инфраструктурный слой.

Такой подход позволяет:
- заменить AI-провайдера без изменения бизнес-логики
- добавить дополнительные AI-сценарии
- корректно обрабатывать недоступность внешнего сервиса через fallback-механизм

### Pydantic

Pydantic используется для:
- валидации входных данных
- описания DTO
- контроля формата данных между слоями приложения

Выбор обусловлен эффективностью и удобством библиотеки, а также тем, что она фактически является частью фреймворка FastAPI.


## Принципы архитектуры
- слоистая архитектура
- изоляция инфраструктуры от бизнес-логики
- инверсия зависимостей через Protocol
- инъекция зависимостей
- управление транзакциями через UoW
- "тонкий" HTTP слой как одна из возможных точек входа

## Упрощенная доменная модель
SQLAlchemy ORM-модель используется как доменная сущность

Мотивы:

- простой домен:
    - нет сложных инвариантов
    - нет переходов между состояниями
- стремление сократить boilerplate код

Компромисс:
 
- домен выражен через ORM-сущность
- но остаётся независимым от БД, сессий и транзакций





# Реализация API
## Описание эндпоинта
```POST /api/contact```

Создание обращения пользователя.

## Пример запроса и ответа

Request:

```
{
  "name": "Ivan",
  "email": "ivan@example.com",
  "phone": "+79999999999",
  "comment": "I have a question about your service"
}
```
Response:
```
{
  "id": 1,
  "status": "accepted"
}
```
## Валидация входных данных

- реализована на границе слоев: shared/dtos.py
- используется Pydantic validation
- проверяется:
    - обязательность полей
    - формат email и телефонного номера
    - длина строк

## Обработка и логирование ошибок
- Ннизкоуровневые ошибки БД/ORM:
    - перехватываются через try / except с сохранением цепочки (from exp)
    - поднимаются доменные исключения, определенные в domen/errors.py
    - маппинг доменных исключений в HTTP ответы реализован в entrypoints/http/exceptions.py через FastAPI exception handlers
    - возвращаемые коды ответов:
        - 409 Conflict
        - 422 Validation Error
        - 500 Internal Server Error
    - логирование ошибок реализовано в entrypoints/http/exceptions.py 

- При перехвате низкоруровых ошибок AI/SMTP клиентов:
    - кастомные исключения не поднимаются
    - низкоруровневые ошибки логируются

- При перехвате ошибок AI клиента также активируется fallback-сценарий


# AI-интеграция
## Назначение

AI используется для автоматического анализа тональности комментариев пользователя.

Пример:

```
Input:

"Мне очень понравился ваш сервис."

AI response:

{
  "sentiment": "positive"
}
```
## AI провайдер

Используется API провайдера Groq.

## Реализация
- Application layer сервис:
    - определен в application/ai/service.py
    - "не знает" о конкретной реализации клиента внешнего API
    - взаимодействует с интерфейсом, который определен в application/interfaces.py ("порт")

- Клиент внешнего API ("адаптер"):
    - определен в infrastructure/ai/groq_client.py
    - обращается к внешнему API через httpx
    - изолирован от application layer
    - соответствует интерфейсу, определенному в application/interfaces.py

- Application сервис получает доступ к внешнему API при сохранении изоляции. Используется инъекция зависимостей через FastAPI Depends.

## Реализация fallback

Если AI сервис недоступен:
- ошибка AI клиента перехватывается
- вызывается модуль, реализующий fallback (infrastructure/ai/fallback.py)
- процесс обработки обращения продолжается

Таким образом недоступность AI не блокирует работу формы обратной связи.

Fallback реализован как функция, которая:
- проверяет наличие в тексте обращения слов из списков с негативным и позитивным значением
- при обнаружении таковых присваивает обращению соответствующую категорию
- при отсутствии таких слов классифицирует обращение как нейтральное
- поддерживает те же категории тональности обращений и формат возвращаемых данных, что и интеграция AI-сервиса

## Использованный промпт

Используется системный prompt:

```
Analyze user message.
Return JSON:
{
    "sentiment": "positive|neutral|negative",
}
```

# Хранение данных

- логи при запуске проекта через Docker Compose хранятся в файле app/logs/app.log
- данные обращений сохраняются в PostgreSQL

- Таблица contacts содержит:
    - имя пользователя
    - email
    - телефон
    - текст обращения
    - результат AI анализа (positive/neutral/negative)
    - источник анализа (ai/fallback)

Миграции управляются через Alembic.
